# 0025 — Canned-response fallback for retro demo

- **Status:** Accepted
- **Date:** 2026-05-04
- **Deciders:** Oliver Kaikane Gate

## Context

The retro demo (`/api/retro`) has several failure modes that aren't bugs:

- **Cost-ceiling blocked** — month-spend hit `ANTHROPIC_MONTHLY_LIMIT_USD`,
  and the route returns 429 (ADR-0010).
- **Rate-limit blocked** — IP hit the 10/hour sliding window, 429
  (ADR-0022).
- **Anthropic API error** — upstream 5xx, transient.
- **Tool-extract / schema-validate failure** — model returned the wrong
  shape; rare, but possible after a prompt change without a snapshot
  update.

A typical error UX would surface a toast ("API call failed, please try
again"). For this surface, that erases the demo's value. The visitor came
to see what a Claude Code retro looks like — showing a toast teaches them
nothing and makes the lab page feel like it's the failure, not the
upstream.

## Decision

**On any failure path that produces a non-2xx response, `/api/retro`
returns the failure envelope alongside a `fallback` object containing a
canned `RetroResponse` keyed to one of three sample-id buckets
(`refactor` / `debug` / `feature`). The client renders the canned response
identically to a live one, with a swapped caption: "API call paused —
showing a sample retrospective." The canned-response shape and the live-
response shape are the same Zod schema, so the renderer doesn't branch.**

Mechanically:

- `lib/canned-retros.ts` carries three hand-curated `RetroResponse`s,
  indexed by `CannedSampleId = "refactor" | "debug" | "feature"`.
- `pickCannedRetro(transcript, sampleMap)` picks the bucket: exact-match
  the transcript against `content/projects.json` sample transcripts, fall
  back to `refactor`. Pure function, no side effects.
- The route emits the fallback on these stages: `ratelimit-blocked`,
  `ceiling-blocked`, `anthropic-call`, `tool-extract`, `schema-validate`.
- The route does **not** emit the fallback on: `validate-input` (visitor
  sent malformed JSON; surface that), `client-init` (server misconfig;
  loud error is correct), `cache-get` / `cache-set` / `ratelimit-check` /
  `ceiling-check` / `cost-log` (infra failures the operator should see).
- Caption swap: live responses get the rust-dot caption "Generated by a
  real Anthropic API call · prompt + transcript only · no persistence";
  fallback responses get the muted-dot caption "API call paused — showing
  a sample retrospective."

## Consequences

**Wins**

- **Demo's value survives failure.** The visitor still sees what a retro
  looks like even when the live path can't run.
- **Honest signal.** The caption swap and dot color tell the visitor
  which path produced the output. Not hidden, not loud.
- **Same renderer.** `RetroOutput` doesn't branch on source — only the
  caption and dot do. One schema, one component, two captions.
- **Defends against low-frequency model drift.** A schema-validate failure
  shows the canned retro instead of an error toast, so the page stays
  intact while Oliver investigates.

**Costs**

- A visitor with no Anthropic key and no budget left could hit "Run
  retro" repeatedly and never know — every response would render fine. The
  caption is the only honest signal. Acceptable: the caption is mono and
  visible; this isn't deceptive UX, it's graceful degradation with
  attribution.
- Three canned retros need to stay current with the design and any
  prompt changes. Today they live in `lib/canned-retros.ts` and Oliver
  edits them by hand. Worth revisiting if more sample transcripts are
  added.

**Deliberately not done**

- **No "demo is paused this month" splash.** The cost ceiling can be
  paused for a month; visitors will still get a canned response. The
  splash would shame the failure; the canned response demonstrates
  through it.
- **No retry-with-backoff on failed Anthropic calls.** A retry burns more
  tokens against the ceiling. The fallback already gives the visitor a
  reasonable response; let them re-click manually if they want to try
  again.
- **No client-side cache of canned responses.** The route emits the
  canned response in the failure envelope; the client renders it
  immediately, no caching needed.

## Alternatives considered

- **Stock error toast on every failure.** Rejected — covered above.
  Erases the demo's value.
- **Server-side simulated response (e.g. stock LLM-style retro from a
  template).** Rejected — would risk the canned response feeling more
  generic than the live one. Hand-curated samples landing inside the
  same component is more honest.
- **Disable the demo entirely when ceiling is hit.** Rejected — the demo's
  page should still tell the story even when the live path is unavailable.
- **Cache the last-good live response per IP and serve it.** Rejected —
  IP attribution is a privacy can of worms (ADR-0010 deliberately doesn't
  attribute by IP), and the cache key already doesn't include IP per
  ADR-0009.
