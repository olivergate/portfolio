# 0037 — Downloadable CV PDF via a dedicated print surface

- **Status:** Accepted
- **Date:** 2026-06-16
- **Deciders:** Oliver Kaikane Gate

## Context

Recruiters expect a PDF CV, but the site is the real artifact — five interactive
pages, a live JD matcher, a rethemeable CV. We want a downloadable PDF that (a)
parses cleanly under 2026 ATS + LLM screening, (b) drives readers back to the site,
and (c) never crosses the project's honesty guardrails (no hidden text, no keyword
stuffing, no prompt injection).

A multi-agent research pass (digested in
`docs/specs/2026-06-04-cv-pdf-distribution.md`) established the load-bearing facts:
clean, single-column, text-extractable PDFs still gate every major ATS; the 2024–26
LLM screening layers read the already-parsed text, not pixels; contact info must sit
in the document body, not a header/footer region; bare-text URLs must accompany any
link; and "75% auto-rejected by ATS" is debunked vendor folklore — the real filter is
the human scan, so a link almost always reaches a person.

The site design is locked (ADR-0002/0006) and the CV page is rethemeable via four
sliders. A PDF needs exactly one canonical theme and an ATS-plain layout — neither of
which the live, rethemeable page is.

## Decision

Ship the PDF as a **build-time snapshot of a dedicated `/cv/print` route**, rendered
with Playwright `page.pdf()` (Plan A), reusing the existing CV components against the
single `content/cv.json` source.

- The route lives **outside the `(site)` route group** — no Nav/Footer/rethemer-FAB/
  scrollspy/JD/Tone/Lab — pinned to `DEFAULT_STYLE`, single-column.
- It is a **deliberate departure from the locked, rethemeable site design**: a new,
  ATS-plain surface, not a re-skin of `/`. Page fidelity to the site's Fraunces/Inter
  typography is kept; the rethemeability is not.
- The CV components are token-driven, so print density is achieved by **overriding
  token values** (a `PRINT_SCALE` merged into the inline style) plus one class-level
  reclaim (`.section-header` padding). Native screen spacing rendered 8 pages; this
  lands a clean **2 pages**, the senior norm.
- A **compact print-only Projects** component (`ProjectsPrint`) replaces the web
  card grid (which carries a meaningless-in-print "Read more →").
- `page.pdf()` (Chromium) emits a **real selectable text layer**; `pdftotext`
  confirms every load-bearing field survives in reading order. `@react-pdf/renderer`
  (Plan B) is rejected — a second layout model and a documented React-19/Next-16
  render-crash risk, for a worse text layer.
- The PDF is generated by `bun run cv:pdf` (`next build` + a self-bootstrapping
  Playwright script) to `public/oliver-gate-cv.pdf`, committed as a static asset.

Privacy split (per the spec's 2026-06-15 decision): the phone lives in `cv.json` but
renders **only in the PDF** (`Header variant="print"`), hidden on the crawlable site;
`/cv/print` is `noindex` + `Disallow`-ed so the phone-bearing HTML source is not
indexed.

## Consequences

- **Wins:** one content source (no PDF/site drift); design-faithful yet ATS-clean;
  excellent extractable text; a self-contained one-command regeneration; the funnel
  (bare URLs + a "paste your role at olivergate.com/jd" line) rides inside the PDF.
- **Costs:** a second layout surface to keep coherent (mitigated — both read
  `cv.json`); the compact print scale is hand-tuned and sits one notch from a 3rd
  page, so adding substantial CV content can re-overflow (regenerate and re-check);
  `page.pdf()` needs a running server at generation time (the script self-manages it).
- **Staleness:** the committed PDF can drift from `cv.json` if regeneration is
  skipped. A CI parse-safety gate (assert key terms survive + ≤2 pages) is the
  intended guard and remains a follow-up.
- **Deliberately not done — and excluded on honesty grounds:** no invisible/white
  keyword text, no metadata keyword stuffing, no prompt-injection strings, no
  fabricated skills. The rejection of these is itself part of the public build story.

## Alternatives considered

- **`@react-pdf/renderer` from `cv.json` (Plan B):** rejected — reimplements CV
  layout in a second model, can't reuse components, finicky fonts (TTF only, no
  WOFF2/variable), and a documented Next-16/React-19 crash risk. Worse text layer
  than `page.pdf()` for no gain.
- **Browser print stylesheet only (no generated asset):** rejected — no stable,
  shareable file to attach to applications or link from the site.
- **Do nothing (site only):** rejected — recruiters and application portals expect a
  PDF; the site alone loses the channel where most screening happens.
