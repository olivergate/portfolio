# doc-lint

A markdown documentation linter that enforces the doc-system conventions
defined in `kit/templates/docs/doc-system/02-style-atlas.md`.

## What it checks

**Hard fails (exit 1):**
- Missing frontmatter block on a categorised doc
- Missing required frontmatter keys per category (e.g., feature docs without `code_paths`)
- ISO date format violations on `date`, `last_verified`, `shipped_on`
- Feature docs missing one of the 11 mandatory section headers
- ADR amendment frontmatter rules (e.g., `authoritative: true` on exactly one practice doc per domain)
- Specs marked `status: shipped` without `shipped_on` set
- Broken cross-references (paths in markdown links, backtick-quoted paths, or bare paths matching the configured `repoPathPrefixes` that don't resolve on disk)
- `AGENTS.md` / `CLAUDE.md` over the configured line limit

**Advisory (exit 0):**
- `last_verified` more than 90 days stale on evergreen docs

## Install

The script is a single TypeScript file with **no external npm dependencies** — it uses only Node built-ins (`fs`, `path`, `child_process`). The only requirement is `tsx` (or any TypeScript runner) to execute it.

### Step 1 — Copy the script

```bash
# From your project root:
mkdir -p tools/doc-lint
cp ~/Documents/Source/retro-claude/kit/tools/doc-lint/doc-lint.ts tools/doc-lint/doc-lint.ts
cp ~/Documents/Source/retro-claude/kit/tools/doc-lint/config.example.json doc-lint.config.json
```

(Or symlink if you want kit upgrades to propagate — but doc-lint config is project-specific, so most teams copy.)

### Step 2 — Add `tsx` as a devDependency

```bash
# pnpm
pnpm add -D tsx

# bun
bun add -d tsx

# npm
npm install --save-dev tsx
```

### Step 3 — Add a script

In your `package.json`:

```json
{
  "scripts": {
    "doc-lint": "tsx tools/doc-lint/doc-lint.ts",
    "doc-lint:changed": "tsx tools/doc-lint/doc-lint.ts --changed"
  }
}
```

### Step 4 — Customise the config

Edit `doc-lint.config.json` (which you copied from `config.example.json`). Key things to tailor:

- **`buckets`** — drop directories your project doesn't use. If you have no `docs/practices/` yet, remove that line; the lint will simply not visit those paths.
- **`repoPathPrefixes`** — must include every root your docs reference. If your project uses `src/` instead of `apps/`, swap them. If you use both, list both. Missing a prefix means cross-ref resolution treats real paths as not-a-repo-path and skips them — silent under-checking.
- **`excludedIndexFiles`** — add any auto-generated index files (regenerated by tooling, not hand-edited) so they only get cross-ref checking.

### Step 5 — Wire as a pre-commit hook (optional but recommended)

```bash
# With husky:
npx husky add .husky/pre-commit "pnpm doc-lint:changed"

# With simple-git-hooks (in package.json):
{
  "simple-git-hooks": {
    "pre-commit": "pnpm doc-lint:changed"
  }
}
```

The `--changed` flag scopes the lint to staged markdown files, keeping pre-commit fast on large repos.

### Step 6 — Wire CI (optional but recommended)

In your CI workflow:

```yaml
- name: Doc lint
  run: pnpm doc-lint
```

The full-tree run takes a few seconds even on large repos.

## CLI flags

```
tsx tools/doc-lint/doc-lint.ts [--changed] [--verbose] [--config <path>]
```

- `--changed` — lint only staged files (for pre-commit). Falls back to full-tree if no staged markdown files.
- `--verbose` — print every file as it's checked.
- `--config <path>` — read config from a custom path. Default: `doc-lint.config.json` at `process.cwd()`.

## Adding a new category check

The category dispatch is in `categorize()` (around line 230) and the per-category `requireKeys` calls are in the `switch` block in `lintFile()`. To add a new category:

1. Add a case to `categorize()` matching the category's path prefix.
2. Add a case to the `switch` in `lintFile()` with `requireKeys(file, fm, [...])` listing the required frontmatter keys.
3. Optionally add per-category checks (date validation, content validation, etc.).

The kit's defaults match the 12 categories in `01-taxonomy.md`. If your project uses a custom category, extend the script (it's a single file, ~600 lines, no dependencies — easy to fork).

## Why a custom linter (and not markdown-lint, alex, etc.)

doc-lint enforces project-specific conventions: frontmatter schemas per category, the 11-section feature-doc template, the `authoritative: true` flag on practice docs, ADR commit-grammar context. Off-the-shelf markdown linters check structure but don't know about these conventions.

The script is small (~600 lines), single-file, and intentionally hand-rolled with no external deps so it can be vendored into any project without dragging in a dependency tree.
